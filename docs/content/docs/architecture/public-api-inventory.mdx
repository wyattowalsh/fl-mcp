This document captures the canonical public MCP interface for FL MCP. External clients may depend
on the resources, templates, prompts, and tools listed here. The generated API reference must
enumerate the same visible roster.

## Public MCP Surface

The normal server exposes one compact, full-power FastMCP surface. It does not expose primitive
tools such as `fl_mixer_set_track_volume` or `mixer_set_track_volume`. The internal FL catalog is
addressed by operation id through `fl_execute`, `fl_batch_execute`, search, schemas, resources, and
workflow prompts.

### Resources

- `runtime://health`
- `runtime://capabilities`
- `providers://matrix`
- `project://snapshot`
- `project://arrangement`

### Resource Templates

- `runtime://capabilities/{domain}`
- `render://jobs/{job_id}`
- `audio://analyses/{analysis_id}`

### Tools

The visible tool roster is exactly 12 tools:

- `fl_status` ã runtime, FL connection, provider, bridge, task, and catalog health.
- `fl_snapshot` ã project/session/domain state read path.
- `fl_search_capabilities` ã search internal operations by intent, domain, provider, safety, and task status.
- `fl_get_capability_schema` ã exact request/response schema, examples, provider support, and safety/readback guidance.
- `fl_execute` ã execute one typed internal FL operation by canonical `domain.operation` id.
- `fl_batch_execute` ã execute ordered operation batches with stop/continue and readback policy.
- `fl_plan` ã preview transaction-envelope changes.
- `fl_apply` ã apply transaction-envelope changes through rollback-aware transaction handling.
- `fl_render` ã render/export task entrypoint.
- `fl_analyze_audio` ã audio-analysis task entrypoint.
- `fl_manage_providers` ã provider lifecycle and routing inspection.
- `fl_browser` ã browser-style plugin, preset, sample, drum-kit, and asset workflow.

`fl_batch_execute`.

## Provider Notes

- `mock` supports the whole catalog and is deterministic for CI and local development.
  In live production mode, `provider="auto"` must not silently fall back to mock;
  callers use `provider="mock"` explicitly for rehearsal.
- `flapi-live` advertises the whole catalog as a forced-live host-file bridge
  path. The bridge attempts candidate FL Python module/function calls for every
  operation. Verified smoke-safe operations report `available`; the remaining
  operations report `attemptable` and can return structured `api_missing`,
  `unsupported_host_behavior`, `path_unavailable`, or `host_exception` failures
  without falling back to mock.
- `piano-roll-script` and `midi-fallback` expose narrower exact operation paths
  where FL Studio host support exists.
- Capability schema responses keep the compatibility `provider_support` list and
  add `provider_support_details` records containing provider, support status,
  execution mode, preconditions, readback operation id, and failure policy.
- Live render/export, audio analysis, sample loading, plugin loading, playlist,
  piano-roll, automation, and other broad DAW flows are attempted through
  `flapi-live` in live mode. They only report success after DAW confirmation.

## Plugin Profile Operations

The `plugins` domain includes declarative profile operations for local plugin
inventory, profile lookup, live parameter probing, raw FL parameter-name
enumeration, raw profile generation, local calibration overlays, priority
support audits, semantic mapped-parameter read/write, preset discovery,
profile-scoped preset loading, and inventory reconciliation. These remain
internal operation ids such as `plugins.enumerate_parameters`,
`plugins.priority_support_audit`, and `plugins.set_mapped_parameter`.

Profiles are semantic seeds, not proof of executable parameter indices. FL
wrapper indices must come from live calibration or a verified local overlay.
Uncalibrated semantic writes return structured failures such as
`calibration_required`, `parameter_unmapped`, `plugin_not_installed`, or
`preset_unavailable`.

Plugin support is priority-scoped: paid installed plugins are P0, detected
paid/pro suite targets are P1, popular/useful stock or free tools are P2, and
the long tail remains P3 inventory-only/on-demand. P3 rows stay discoverable,
but they are not release blockers.

## Migration Notes

Clients that previously called primitive visible tools must migrate to the compact executor:

```json
{
  "tool": "fl_execute",
  "arguments": {
    "operation_id": "mixer.set_track_volume",
    "request": { "track_index": 0, "volume": 0.75 },
    "provider": "mock",
    "readback": {
      "operation_id": "mixer.get_track_volume",
      "request": { "track_index": 0 },
      "provider": "mock"
    }
  }
}
```

Use this agent loop:

1. `fl_status` or `fl_snapshot`
2. `fl_search_capabilities`
3. `fl_get_capability_schema`
4. `fl_execute`, `fl_batch_execute`, `fl_plan`, or `fl_apply`
5. Readback or snapshot verification

## Governance

Any addition, removal, or reorganization of the public agent surface is a public API change:

- update this inventory,
- update `docs/content/docs/release-notes.mdx`,
- update the relevant `goals/` package when the change belongs to an active goal,
- regenerate generated reference docs with `pnpm --dir docs --ignore-workspace docs:generate-reference`,
- verify generated reference docs with `pnpm --dir docs --ignore-workspace docs:verify-reference`,
- run `uv run pytest tests/contract/test_public_surface.py`,
- inspect the FastMCP roster with `uv run fastmcp inspect fastmcp.json --format mcp`.

Generated reference files under `docs/content/docs/reference/generated/` must be regenerated, not
hand-edited.

See [Validation Runbook](/docs/validation-runbook) for command groupings and
evidence expectations.
