# Contributing to FL-MCP

Thank you for considering a contribution to FL-MCP. This guide covers everything
you need to get started.

## Prerequisites

| Tool | Version | Purpose |
|------|---------|---------|
| Python | 3.12+ | Runtime |
| [uv](https://docs.astral.sh/uv/) | latest | Package & virtualenv management |
| git | 2.x+ | Version control |

## Setup

```bash
git clone https://github.com/wyattowalsh/fl-mcp.git
cd fl-mcp
uv sync --all-extras --dev
```

Copy the environment template if you need to override defaults:

```bash
cp .env.example .env
```

## Development workflow

1. Create a feature branch from `main`:
   ```bash
   git checkout -b feat/my-feature main
   ```
2. Make your changes in a vertical slice that compiles and tests.
3. Run the checks described below.
4. Commit using [Conventional Commits](#commit-messages) and open a PR.

## Testing

Run the full test suite (excluding benchmarks):

```bash
uv run pytest --ignore=tests/benchmarks
```

Run a single suite:

```bash
uv run pytest tests/unit
uv run pytest tests/contract
uv run pytest tests/integration
```

Run with coverage:

```bash
uv run pytest --cov=fl_mcp --cov-report=term-missing --ignore=tests/benchmarks
```

## Linting

```bash
uv run pre-commit run --all-files --show-diff-on-failure
uv run ruff check src/ tests/
uv run ruff format --check src/ tests/
```

Auto-fix what ruff can:

```bash
uv run ruff check --fix src/ tests/
uv run ruff format src/ tests/
```

## Type checking

```bash
uv run ty check src/
```

## Local quality gate

Use the same broad checks as CI before opening a PR:

```bash
just ci
```

If you do not have `just`, run the component gates directly:

```bash
uv run pre-commit run --all-files --show-diff-on-failure
uv run ty check src/
uv run pytest -q
pnpm --dir docs --ignore-workspace docs:verify-reference
pnpm --dir docs --ignore-workspace lint
pnpm --dir docs --ignore-workspace check
pnpm --dir docs --ignore-workspace build
uv build
uv run --with twine==6.2.0 twine check dist/*
```

## Commit messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

- `feat:` -- new feature
- `fix:` -- bug fix
- `docs:` -- documentation only
- `test:` -- adding or updating tests
- `refactor:` -- code change that neither fixes a bug nor adds a feature

## Pull request process

1. Describe **what** changed and **why**.
2. Reference related issues (e.g. `Closes #42`).
3. Ensure all CI checks pass (tests, lint, typecheck, docs build, package build).
4. Keep the diff focused; unrelated changes belong in a separate PR.

## Release notes and changelog

Release notes are generated by GitHub from merged pull requests and labels using
`.github/release.yml`. Use labels such as `feature`, `bug`, `docs`, `ci`,
`release`, `tooling`, `dependencies`, and `tests` so the generated changelog
lands in the right section. Use `skip-changelog` only for changes that should be
excluded from generated release notes.

Maintainers can preview the generated changelog with the **Changelog Preview**
workflow before publishing a tag or prerelease.

## Code style

- Follow existing patterns in the codebase.
- Keep the public MCP surface intentionally small.
- Prefer resources-first reads over tool-based reads.
- Preserve one canonical project graph and one canonical transaction envelope.
- Domain schemas are strongly typed; every mutation declares a rollback/checkpoint class.
- Keep docs and generated references in sync with code changes.
